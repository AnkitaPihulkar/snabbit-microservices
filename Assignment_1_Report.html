B. Brief Application Description

Snabbit is India's first 10-minute on-demand home services application focusing on providing quick, reliable, and verified female house help (Experts) to users. The platform connects urban households looking for immediate assistance with trained, background-verified professionals at their doorsteps.

Unlike monthly agency models, Snabbit operates on a transparent, hourly-based pricing structure with flexible, contract-free booking options. Services offered include kitchen cleaning, dishwashing, fan/window cleaning, laundry, and bathroom cleaning.

C. Architecture of the Application

The application follows the Database-per-Microservice pattern, implemented as Node.js/Express services containerized in independent Docker containers. Inter-service boundaries are bridged by an API Gateway acting as a single public entry point.

D. Service Decomposition Rationale

Decomposition was completed using the Decomposition by Sub-Domain (Domain-Driven Design - DDD) approach:

  • Customer Domain (Core Subdomain): Manages user profiles, addresses, and contacts. Changes to customer attributes are isolated to this service.
  • Expert Domain (Core Subdomain): Manages the professional registry, Aadhaar verification state, and active availability status (e.g. available, busy).
  • Booking Domain (Supporting Subdomain): Orchestrates the transaction lifecycle, scheduling coordinates, and cost calculations.

Each service implements the Single Responsibility Principle (SRP): a service has only one reason to change, which decouples development lifecycles and enables independent deployments.

E. Service API Design

1. Customer Service

  • POST /v1/customers: Register a customer.
    Payload: { "name": "Shreyash", "email": "shreyash@bits.edu.in", "phone": "9999999999", "address": "Flat 302, Bangalore" }
  • GET /v1/customers/:id: Get customer profile details.

2. Expert Service

  • POST /v1/experts: Register a new expert.
    Payload: { "name": "Priya Sharma", "phone": "9876543210", "skills": "kitchen cleaning", "aadhaar": "123456789012", "hourly_rate": 150, "city": "Mumbai" }
  • PUT /v1/experts/:id/status: Update expert status (e.g., to busy or available).
  • GET /v1/experts?city=Mumbai&status=available: Query available experts.

3. Booking Service

  • POST /v1/bookings: Create a booking.
    Payload: { "customer_id": 1, "expert_id": 1, "service_type": "kitchen cleaning", "duration_hours": 3, "booking_date": "2026-07-20" }
  • PUT /v1/bookings/:id/status: Complete or cancel a booking.

4. API Gateway

  • Proxies all /api/v1/customers/*, /api/v1/experts/*, and /api/v1/bookings/* requests.
  • GET /api/v1/bookings/:id/detail: **API Composition endpoint**. Aggregates booking details with customer profile and expert profile data in a single client response.

F. Database Design & Pattern Used

We utilized the Database per Microservice pattern. Each service owns its data storage:

  • Customer Service owns customer.db
  • Expert Service owns expert.db
  • Booking Service owns booking.db

SQLite was selected as the database technology. SQLite files run inside the respective container’s persistent volumes. This design provides high file-based write speeds, zero networking configuration overhead for simple demos, and guarantees that database schemas cannot be accessed or coupled across microservice boundaries.

G. Communication Style between Services

Inter-service communication is designed as synchronous request-response over HTTP REST:

  • When a booking is requested, the Booking Service makes synchronous REST calls to the Customer Service (to verify identity) and the Expert Service (to verify availability and skills).
  • Once verified, the Booking Service makes a synchronous PUT call to the Expert Service to change their state to busy, completing the local transaction.

This synchronous check-then-act flow guarantees data integrity during matching and releases resources when bookings are completed.

H. API Gateway & API Composition

The **API Gateway** acts as the single entry point. It handles:

  1. Request Routing: Hides internal ports (5001, 5002, 5003) and routes public requests under unified /api/v1/ namespaces.
  2. API Composition: Instead of forcing the client to make three sequential HTTP calls (booking details, customer details, and expert details), the gateway exposes /api/v1/bookings/:id/detail, calls the three services concurrently via Axios, merges the responses, and returns a unified payload.

I. API Versioning Strategy

We use **URI-based API Versioning** (e.g., /api/v1/). Major version releases prevent breaking changes for external client integrations. Non-breaking extensions (such as adding optional fields) maintain the current prefix. We also tag build artifacts and Docker images using **Semantic Versioning (SemVer)** (e.g., v1.0.0).

J. Docker Containerization Steps

All services contain independent Dockerfiles. Deployment is orchestrated via Docker Compose.
